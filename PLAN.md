# Implementation Plan: template-fullstack-kubernetes

## Context

### Goal
Boosha needs a standardized, reusable template for fullstack projects — a Python backend API paired with a React frontend SPA — deployed on Kubernetes via Helm. Any new fullstack project that targets Kubernetes should be bootstrapped from this template with working CI/CD, hardened Docker images, a production-ready Helm chart, pre-commit hooks, and security scanning — out of the box.

This is the third template in a family of three. The other two are:
- **template-backend-python** — standalone Python/FastAPI backend
- **template-frontend-react** — standalone React/Vite SPA

This template combines both into a single monorepo and adds what makes a fullstack Kubernetes deployment real: the services **communicate** (the React SPA calls the FastAPI backend), the Helm chart deploys both as a coordinated stack, and the CI pipeline validates the integration end-to-end.

### What this template must deliver
A **standalone repository** that someone can clone/use as a GitHub template to start a new fullstack project. The pipeline lives at `.github/workflows/ci.yml` (root level) and **must be verified green** before the template is considered done.

### What makes this different from the two standalone templates

| Standalone templates | This template |
| --- | --- |
| Frontend uses in-memory state for items | Frontend **fetches from the backend API** (`/api/items`) |
| Each has its own independent CI | CI builds **both** services, plus an **integration test** that proves they work together |
| No orchestration | **Helm chart** deploys both services on Kubernetes, with hardened security contexts |
| No inter-service networking | nginx reverse-proxies `/api/*` to the backend Service; Vite dev proxy handles it in development |
| Single Dockerfile | Two Dockerfiles + **docker-compose** for local development |

**Conventions:**
- **Monorepo with isolated stacks:** `backend/` and `frontend/` are self-contained — each has its own package manager config, Dockerfile, lockfile, and test suite. A developer working only on the backend doesn't need pnpm; a developer working only on the frontend doesn't need uv.
- **Feature folders (frontend):** Pages and components grouped by feature (`pages/Items/`, `components/Header/`), each folder containing the component and its CSS module.
- **Barrel-free (frontend):** No `index.ts` re-exports. Imports use explicit file paths. Barrel files cause circular dependency issues and make tree-shaking harder.
- **Helm chart at root level:** `helm-chart/` sits alongside the services, not nested inside either. It's an infrastructure concern that spans both.
- **Root-level `.pre-commit-config.yaml`:** A single pre-commit config at root orchestrates hooks for both stacks. Hooks that run on backend files use `files:` patterns scoped to `backend/`; frontend hooks are scoped to `frontend/`.
- **Root-level `docker-compose.yml`:** Orchestrates both services for local development. Lives at root because it references both `backend/Dockerfile` and `frontend/Dockerfile`.

## Chapter 1: Backend — Project Scaffolding

### Type
feat

### What
Create the backend application inside `backend/`. This is parallel to template-backend-python Step 1, with one critical addition: **CORS middleware** so the frontend (running on a different origin in development) can call the API.

**Files to create:**
- `backend/.gitignore`
- `backend/pyproject.toml`
- `backend/src/__init__.py`, `backend/src/main.py`, `backend/src/models.py`
- `backend/tests/__init__.py`, `backend/tests/test_health.py`, `backend/tests/test_items.py`

#### `backend/.gitignore` requirements
- Python-specific: `.venv/`, `__pycache__/`, `*.pyc`
- Tool caches: `.mypy_cache/`, `.pytest_cache/`, `.ruff_cache/`
- Coverage: `htmlcov/`, `coverage.xml`, `.coverage`
- Build: `dist/`, `*.egg-info/`
- Secrets: `.env`

#### `backend/pyproject.toml` requirements
- **Package manager:** uv
- **Python version:** >=3.12
- **Core dependencies:** FastAPI, uvicorn[standard]
- **Dev dependencies:** ruff, mypy, pytest, pytest-cov, httpx
- **Ruff config:**
  - `line-length = 88`, `target-version = "py312"`
  - Lint rules: `E`, `F`, `I`, `W`, `UP`, `B`, `SIM` (covers pycodestyle, pyflakes, isort, pyupgrade, bugbear, simplify)
  - isort: `known-first-party = ["src"]`
- **Mypy config:** `ignore_missing_imports = true`
- **Pytest config:** `testpaths = ["tests"]`

**Why ruff:** Replaces black (formatting), flake8 (linting), and isort (import sorting) with a single Rust-based tool. 10-100x faster, simpler config, one tool to maintain instead of three.

#### `backend/src/main.py` requirements
Must go beyond a hello-world. It must demonstrate:
- A GET `/health` endpoint (for container probes and smoke tests)
- A POST `/items` endpoint with Pydantic input validation
- A GET `/items/{item_id}` endpoint with path parameter
- Proper error handling (404 for missing items, 422 auto-handled by Pydantic/FastAPI)
- Simple in-memory storage (dict) — no database, keeps the template self-contained
- **CORS middleware** (`fastapi.middleware.cors.CORSMiddleware`) configured with:
  - `allow_origins` including `http://localhost:5173` (Vite dev server) and configurable via environment variable for production
  - `allow_methods: ["*"]`, `allow_headers: ["*"]`
  - This is what allows the React frontend (running on a different port/origin) to call the API. Without CORS, the browser blocks cross-origin requests.

#### `backend/src/models.py` requirements
- Pydantic models: `ItemCreate` (input — name + price), `Item` (response — id + name + price)
- Must demonstrate Pydantic v2 usage with FastAPI

#### Test requirements
- `test_health.py`: Health endpoint returns 200 + expected JSON
- `test_items.py`: POST with valid payload (201), POST with invalid payload (422), GET existing item (200), GET non-existing item (404)
- Use `fastapi.testclient.TestClient` (backed by `httpx`)
- Tests must exercise the CRUD round-trip: create an item, then fetch it by the returned id and verify the data matches

# 3. Lint check

## Chapter 3: Backend Dockerfile

### Type
build

### What
Create a production-ready, hardened Dockerfile for the backend inside `backend/`. Parallel to template-backend-python Step 2.

**Files:**
- `backend/Dockerfile`
- `backend/.dockerignore`

**Multi-stage build (2 stages):**

**Stage 1 — `builder`:** Installs `uv` (pinned with digest SHA) and production dependencies. The `uv`/`uvx` binaries and any build tooling stay in this stage only.

**Stage 2 — `runtime`:** Copies only the installed venv and source from `builder`. No `uv`, no build tools — smaller image, smaller attack surface.

**Requirements:**

1. **Digest SHA base image pinning** — e.g., `python:3.12.13-slim@sha256:<digest>` (multi-arch manifest list digest, not platform-specific). The human-readable tag stays for readability, the `@sha256:` suffix is what Docker resolves. Dependabot (Chapter 12) with `package-ecosystem: docker` opens PRs when a new digest is available.

2. **Digest SHA uv image pinning** — Same approach for `ghcr.io/astral-sh/uv`: pin to a specific version tag + digest SHA (not `:latest`).

3. **Runtime environment variables:**
   - `PYTHONDONTWRITEBYTECODE=1` — No `.pyc` files (not useful in containers)
   - `PYTHONUNBUFFERED=1` — Unbuffered stdout/stderr for proper logging
   - `VIRTUAL_ENV=/app/.venv` + `PATH` — Explicit venv path, avoids relying on implicit `.venv` location

4. **Non-root user** — Create a dedicated user `appuser` (UID 1000) and switch to it before CMD. Required for the Helm chart's `runAsNonRoot: true` (Chapter 6). Follows the Principle of Least Privilege: a container escape no longer gives root on the host.

5. **File ownership** — `COPY --from=builder --chown=appuser:appuser /app /app` so files are owned by the non-root user, not root.

6. **Layer optimization** — Copy `pyproject.toml` + `uv.lock` and install deps before copying source code, so dependency install is cached across builds.

7. **No dev dependencies in production** — Use `uv sync --no-dev --no-install-project --frozen`

8. **HEALTHCHECK** — `urllib.request.urlopen` against `/health` with Python-level `timeout=2`. Docker-level `--timeout=5s` acts as a safety net. Lets Docker and orchestrators (Compose, ECS) monitor container health.

9. **Expose port 8000** — Run uvicorn on `0.0.0.0:8000`

## Chapter 4: Frontend Dockerfile

### Type
build

### What
Create a production-ready, hardened Dockerfile for the frontend inside `frontend/`, plus the nginx configuration that serves the SPA and reverse-proxies API calls to the backend. Parallel to template-frontend-react Step 4.

**Files:**
- `frontend/Dockerfile`
- `frontend/.dockerignore`
- `frontend/nginx.conf`

**Multi-stage build (2 stages):**

**Stage 1 — `builder`:** Installs pnpm via corepack and builds the production bundle. The Node runtime, pnpm, `node_modules/`, and all build tooling stay in this stage only.

**Stage 2 — `runtime`:** Copies only the built `dist/` directory and the nginx config from `builder`. No Node, no pnpm, no source code — just static files and nginx.

**Requirements:**

1. **Digest-pinned base images** — e.g., `node:22-slim@sha256:<digest>` for the builder, `nginx:stable-alpine@sha256:<digest>` for the runtime. Dependabot (Chapter 12) opens PRs when new digests are available.

2. **pnpm installation in builder:**
   - Enable corepack: `corepack enable` (ships with Node 22, manages pnpm version from `package.json`'s `packageManager` field)
   - `corepack prepare --activate` — installs the exact pnpm version declared in `package.json`
   - Single source of truth for pnpm version is `package.json` — no separate pin in the Dockerfile

3. **Layer optimization:**
   - Copy `package.json` + `pnpm-lock.yaml` first and run `pnpm install --frozen-lockfile`
   - Then copy the rest of the source and run `pnpm build`
   - Dependency install is cached across builds when only source code changes

4. **No dev dependencies in production:** Dev dependencies are needed at build time (TypeScript, Vite, etc.), so `pnpm install --frozen-lockfile` installs everything in the builder stage. But the runtime stage only receives `dist/` — no `node_modules/` at all.

5. **Runtime environment:**
   - `nginx:stable-alpine` — small image (~10MB), only serves static files
   - Copy `dist/` to `/usr/share/nginx/html/`
   - Copy `nginx.conf` to `/etc/nginx/conf.d/default.conf`

6. **Non-root user:**
   - nginx on Alpine runs as `nginx` user (UID 101) when configured properly
   - `nginx.conf` uses `listen 8080` (unprivileged port), writes pid to `/tmp/nginx.pid`, logs to stdout/stderr
   - Switch to `USER nginx` before CMD
   - Satisfies Kubernetes `runAsNonRoot: true` policies (Chapter 6)

7. **HEALTHCHECK:**
   - `HEALTHCHECK --interval=30s --timeout=5s --start-period=5s --retries=3 CMD wget -qO- http://localhost:8080/health || exit 1`
   - Uses `wget` (available in Alpine) instead of `curl` (not installed by default in Alpine)

8. **Expose port 8080** — unprivileged port (not 80, which requires root)

#### `frontend/nginx.conf` requirements

```nginx
server {
    listen 8080;
    server_name _;
    root /usr/share/nginx/html;
    index index.html;

    # Health endpoint for Docker HEALTHCHECK and orchestrator probes
    location = /health {
        access_log off;
        return 200 '{"status":"healthy"}';
        add_header Content-Type application/json;
    }

    # API reverse proxy — forwards /api/* to the backend service
    # In production (Kubernetes): backend is reachable via its Service name
    # The backend URL is configurable via BACKEND_URL env var (set in ConfigMap / docker-compose)
    location /api/ {
        proxy_pass http://backend:8000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # SPA fallback: serve index.html for all routes that don't match a file
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache static assets aggressively (Vite adds content hashes to filenames)
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
}
```

Key decisions:
- **`/api/` reverse proxy** (`proxy_pass http://backend:8000/`) — the critical fullstack piece. In Kubernetes, `backend` resolves to the backend Service via DNS. In Docker Compose, `backend` resolves to the backend container via Docker networking. The trailing `/` in `proxy_pass` strips the `/api/` prefix: a request to `/api/items` becomes `/items` on the backend. This matches the Vite dev proxy behavior from Chapter 2.
- **SPA fallback** (`try_files $uri $uri/ /index.html`) — required for client-side routing. Without this, direct navigation to `/items` returns a 404.
- **`/health` endpoint** — returns `{"status":"healthy"}` with `application/json`. Consumed by Docker HEALTHCHECK, Kubernetes probes, and CI smoke tests. `access_log off` prevents health check spam.
- **Asset caching** — Vite outputs files like `assets/index-abc123.js` with content hashes. Safe to cache forever (`immutable`).
- **Security headers** — no iframe embedding (clickjacking), no MIME sniffing, controlled referrer. Starting point; downstream projects may add CSP, HSTS.

**Note on `proxy_pass` target configurability:** The hardcoded `http://backend:8000` works for both Docker Compose (service name `backend`) and Kubernetes (Service name `backend`). If the backend Service is named differently, users change `values.yaml` and the ConfigMap renders the correct upstream. For more advanced setups (e.g., multiple backends, path-based routing), the nginx.conf can be templated via a ConfigMap volume mount — documented in the README as a customization option.

**Note on Alpine vs Debian-slim for nginx:** Alpine produces a significantly smaller runtime image (~10MB vs ~70MB). The trade-off is musl vs glibc, but for a static file server this is irrelevant.

#### `frontend/.dockerignore` requirements
- `node_modules/` — pnpm install runs inside the builder
- `.git/` — no git history in context
- `dist/` — prevent stale local builds from leaking into the image
- `coverage/` — test artifacts
- `.env`, `.env.*` — secrets
- `*.md` — docs not needed in image
- `.vscode/`, `.idea/` — editor configs

## Chapter 5: Docker Compose — Local Development

### Type
build

### What
Create a `docker-compose.yml` at the repo root that lets developers run the full stack locally with a single command. This is a new chapter — the standalone templates don't have Docker Compose because they are single-service.

**Files:**
- `docker-compose.yml`

#### Why Docker Compose in a Kubernetes template

Docker Compose and Helm serve different purposes and **both belong** in this template:

| | Docker Compose | Helm |
| --- | --- | --- |
| **Where** | Developer laptop | Kubernetes cluster |
| **Purpose** | Quick local run of the full stack | Production deployment |
| **Networking** | Docker bridge network (service names resolve automatically) | Kubernetes Services + DNS |
| **Config** | `docker-compose.yml` | `values.yaml` + templates |
| **When to use** | "I want to see both services running together in 30 seconds" | "I'm deploying to staging/prod" |

A developer should not need a Kubernetes cluster to test the full stack locally. Docker Compose gives them a production-like environment (both containers, real networking, same Dockerfiles) without kubectl or minikube.

#### `docker-compose.yml` requirements

```yaml
services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:8000/health', timeout=2)"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 5s

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    depends_on:
      backend:
        condition: service_healthy
```

Key decisions:

- **Service names match Kubernetes:** The backend service is named `backend`, which is the same hostname that `nginx.conf`'s `proxy_pass http://backend:8000/` expects. Docker Compose creates a default network where services resolve each other by name. This means the **same `nginx.conf`** works in both Docker Compose and Kubernetes — no environment-specific config needed.

- **`depends_on` with health condition:** The frontend container waits until the backend reports healthy before starting. This prevents nginx from returning 502 errors on `/api/*` during startup. The backend's HEALTHCHECK (defined in its Dockerfile, Chapter 3) is what Docker Compose monitors.

- **Production Dockerfiles, not dev overrides:** The compose file builds using the production Dockerfiles. This is intentional — it validates that the same images that CI builds and Helm deploys work correctly together. For developers who want hot reload during active development, the README (Chapter 14) documents the alternative: run `uvicorn --reload` and `pnpm dev` natively, without Docker.

- **No volume mounts for source code:** Volume mounts (e.g., `./backend/src:/app/src`) would enable hot reload inside containers, but they introduce platform-specific issues (file watching on macOS/Windows via Docker Desktop is slow and unreliable, especially for large `node_modules`). The compose file stays simple and production-like. Developers who want fast iteration use native tools directly.

- **No `docker-compose.override.yml`:** A single file keeps things simple. If a downstream project needs dev-specific overrides (environment variables, debug ports, volume mounts), they add their own override file — the README documents how.

- **Ports:** Backend on `8000` (same as standalone), frontend on `8080` (same as the nginx container). Both are accessible from the host for debugging.

#### 6b. Backend Deployment

**`templates/backend-deployment.yaml`:**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "fullstack-app.backend.fullname" . }}
  labels:
    {{- include "fullstack-app.backend.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.backend.replicaCount }}
  selector:
    matchLabels:
      {{- include "fullstack-app.backend.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "fullstack-app.backend.selectorLabels" . | nindent 8 }}
    spec:
      serviceAccountName: {{ include "fullstack-app.backend.fullname" . }}
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        fsGroup: 1000
      containers:
        - name: backend
          image: "{{ .Values.backend.image.repository }}:{{ .Values.backend.image.tag | default .Chart.AppVersion }}"
          imagePullPolicy: {{ .Values.backend.image.pullPolicy }}
          ports:
            - name: http
              containerPort: 8000
              protocol: TCP
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop: [ALL]
          startupProbe:
            httpGet:
              path: {{ .Values.backend.probes.startup.path }}
              port: http
            failureThreshold: {{ .Values.backend.probes.startup.failureThreshold }}
            periodSeconds: {{ .Values.backend.probes.startup.periodSeconds }}
          livenessProbe:
            httpGet:
              path: {{ .Values.backend.probes.liveness.path }}
              port: http
            periodSeconds: {{ .Values.backend.probes.liveness.periodSeconds }}
            failureThreshold: {{ .Values.backend.probes.liveness.failureThreshold }}
          readinessProbe:
            httpGet:
              path: {{ .Values.backend.probes.readiness.path }}
              port: http
            periodSeconds: {{ .Values.backend.probes.readiness.periodSeconds }}
            failureThreshold: {{ .Values.backend.probes.readiness.failureThreshold }}
          resources:
            {{- toYaml .Values.backend.resources | nindent 12 }}
```

Key points:
- **`securityContext` at pod level:** `runAsNonRoot: true` + `runAsUser: 1000` + `fsGroup: 1000`. Kubernetes will refuse to start the pod if the container image's default user is root. This pairs with the `USER appuser` (UID 1000) in the backend Dockerfile (Chapter 3).
- **`securityContext` at container level:** `allowPrivilegeEscalation: false` + `readOnlyRootFilesystem: true` + `capabilities.drop: [ALL]`. This is the Principle of Least Privilege in practice — the container cannot gain additional privileges, cannot write to its filesystem (only to mounted volumes if any), and has zero Linux capabilities.
- **`startupProbe`:** Allows the container up to 300 seconds (30 × 10s) to start before Kubernetes considers it failed. This is critical for applications with slow warmup (e.g., ML model loading). Without `startupProbe`, `livenessProbe` would kill the pod during warmup.
- **`livenessProbe` vs `readinessProbe`:** Liveness checks if the process is alive (restart if not). Readiness checks if it can serve traffic (remove from Service endpoints if not). Both hit `/health` but have different failure consequences.
- **Image tag defaults to `.Chart.AppVersion`:** See 6g.

#### 6c. Frontend Deployment

**`templates/frontend-deployment.yaml`:**

Same structure as the backend deployment, with these differences:
- **`runAsUser: 101`** (nginx user UID on Alpine, matching the `USER nginx` in the frontend Dockerfile, Chapter 4)
- **`fsGroup: 101`**
- **`containerPort: 8080`** (nginx's unprivileged port)

**Why the `/tmp` volume:** `readOnlyRootFilesystem: true` means the container cannot write anywhere except explicitly mounted volumes. nginx writes its pid to `/tmp/nginx.pid` (configured in Chapter 4's nginx.conf for non-root operation). Without this mount, nginx fails to start. The `emptyDir` is backed by the node's tmpfs — it's fast, ephemeral, and doesn't persist across pod restarts.

#### 6d. Services

**`templates/backend-service.yaml`** and **`templates/frontend-service.yaml`:**

Both are `ClusterIP` Services (internal-only, no external exposure). They route traffic to pods matched by selector labels.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: {{ include "fullstack-app.backend.fullname" . }}
  labels:
    {{- include "fullstack-app.backend.labels" . | nindent 4 }}
spec:
  type: {{ .Values.backend.service.type }}
  ports:
    - port: {{ .Values.backend.service.port }}
      targetPort: http
      protocol: TCP
      name: http
  selector:
    {{- include "fullstack-app.backend.selectorLabels" . | nindent 4 }}
```

**Why the backend Service name matters:** The frontend's `nginx.conf` uses `proxy_pass http://backend:8000/`. In Kubernetes, `backend` resolves via cluster DNS to the backend Service's ClusterIP — but only if the Service is named `backend`. The `_helpers.tpl` generates names like `<release>-backend`. To keep nginx.conf simple and portable (same config in Docker Compose and K8s), the ConfigMap (6f) can inject the actual backend hostname. Alternatively, the chart's NOTES.txt can document that the default `nginx.conf` assumes the backend Service is reachable as `backend` — which is true when the Helm release name is the chart name, or when using the ConfigMap-based approach.

**Practical approach:** The `nginx.conf` in the frontend image uses `backend` as the upstream hostname. This works in Docker Compose out of the box. For Kubernetes, the backend Service is named via `_helpers.tpl` (e.g., `fullstack-app-backend`). To bridge this, the ConfigMap (6f) provides the backend URL, and the frontend Deployment mounts a templated `nginx.conf` snippet via ConfigMap volume — or more simply, the backend Service includes an alias via `externalName` or the chart names the backend Service simply `backend` within the namespace. The simplest approach: the `_helpers.tpl` generates a short name for the backend Service (just `backend` when the release name is omitted), or the frontend Deployment includes an `initContainer` or environment variable that the nginx config reads via `envsubst`. Document the trade-offs in the README; default to the simplest working approach.

**Recommended simplest approach:** Use Helm's templating to generate the `nginx.conf` as a ConfigMap, replacing `backend` with the actual backend Service name:

```yaml
# templates/configmap.yaml includes:
nginx-upstream.conf: |
  upstream backend_upstream {
    server {{ include "fullstack-app.backend.fullname" . }}:{{ .Values.backend.service.port }};
  }
```

And the nginx `location /api/` block references `backend_upstream`. This is covered in 6f.

#### 6e. Security hardening (Principle of Least Privilege)

This section summarizes the security posture applied across the chart. Every choice follows the Principle of Least Privilege: each component has **only the permissions strictly necessary for its job**.

**Pod-level `securityContext`** (applied to both deployments):
| Field | Backend | Frontend | Why |
| --- | --- | --- | --- |
| `runAsNonRoot` | `true` | `true` | Kubernetes refuses to start the pod if the image runs as root. Pairs with `USER` in Dockerfile. |
| `runAsUser` | `1000` | `101` | Must match the `USER` in the Dockerfile. Backend: `appuser` (1000). Frontend: `nginx` (101). |
| `fsGroup` | `1000` | `101` | Files created by the pod are owned by this group. Matches the user. |

**Container-level `securityContext`** (applied to both containers):
| Field | Value | Why |
| --- | --- | --- |
| `allowPrivilegeEscalation` | `false` | Prevents the process from gaining more privileges than its parent (blocks `setuid` binaries, `ptrace`, etc.). |
| `readOnlyRootFilesystem` | `true` | The container cannot write to its own filesystem. Forces explicit volume mounts for any writable paths (e.g., `/tmp` for nginx). Limits what an attacker can do after a compromise. |
| `capabilities.drop` | `[ALL]` | Removes all Linux capabilities (NET_RAW, SYS_ADMIN, etc.). The container has zero special kernel privileges. If a specific capability is needed, it must be explicitly added back — and that decision is auditable. |

**Why this matters:** Without these settings, a compromised container runs as root with full capabilities. An attacker who escapes the container has root on the node. With these settings, the same attacker is in a sandbox: non-root, read-only filesystem, zero capabilities. Breaking out requires chaining a second vulnerability — the security context raises the bar significantly.

**Compatibility with Pod Security Admission (PSA):** Kubernetes 1.25+ enforces Pod Security Standards via the built-in PSA admission controller. The `restricted` profile (the strictest) requires exactly the settings above. Charts without these settings are **rejected** on clusters with `restricted` enforcement — the pod never starts. This chart is `restricted`-compliant out of the box.

#### 6f. ConfigMap

**`templates/configmap.yaml`:**

The ConfigMap serves two purposes:

1. **Backend URL for nginx:** Provides the backend Service's DNS name so the frontend's nginx knows where to proxy `/api/*` requests. This bridges the gap between Docker Compose (where `backend` resolves by service name) and Kubernetes (where the full Service name depends on the Helm release).

2. **Application-level configuration:** A place for shared environment variables that both services may need (e.g., `APP_ENV`, feature flags). Downstream projects extend this as needed.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "fullstack-app.fullname" . }}
  labels:
    {{- include "fullstack-app.labels" . | nindent 4 }}
data:
  BACKEND_URL: "http://{{ include "fullstack-app.backend.fullname" . }}:{{ .Values.backend.service.port }}"
```

The frontend Deployment uses this ConfigMap to configure the nginx upstream dynamically. Two approaches (choose one during implementation):

**(a) `envsubst` at container startup (simpler):** The nginx.conf uses `${BACKEND_URL}` as the proxy target. The frontend container's `CMD` runs `envsubst` to replace the variable before starting nginx. This is a common pattern for dynamic nginx config in containers.

**(b) ConfigMap-mounted nginx snippet (more Kubernetes-native):** The ConfigMap includes a rendered nginx upstream block. The frontend Deployment mounts it as a volume at `/etc/nginx/conf.d/upstream.conf`. nginx's `include` directive loads it.

Approach (a) is recommended for simplicity. The Dockerfile's CMD becomes:
```dockerfile
CMD ["/bin/sh", "-c", "envsubst '${BACKEND_URL}' < /etc/nginx/conf.d/default.conf.template > /etc/nginx/conf.d/default.conf && nginx -g 'daemon off;'"]
```

And the nginx.conf (Chapter 4) changes `proxy_pass http://backend:8000/` to `proxy_pass ${BACKEND_URL}/`.

#### 6g. Image tag strategy

**No `tag: latest`.** This is an anti-pattern in production Kubernetes for three reasons:
1. **Rollback is guesswork:** `latest` points to different images over time — you can't roll back to "the latest from yesterday".
2. **`imagePullPolicy: IfNotPresent` + `latest` is dangerous:** Nodes cache images. After pushing a new `latest`, some pods get the new image, some keep the old cached one. Different nodes run different code.
3. **No traceability:** `kubectl describe pod` shows `latest` — no way to trace back to a specific commit.

**Default to `.Chart.AppVersion`:**
```yaml
image: "{{ .Values.backend.image.repository }}:{{ .Values.backend.image.tag | default .Chart.AppVersion }}"
```

In `values.yaml`, `tag` is set to `""` (empty string). When empty, `default .Chart.AppVersion` kicks in, using the version from `Chart.yaml`. This is bumped when releasing a new version of the application.

**Override with commit SHA for CI/CD deployments:**
```bash
helm upgrade myapp ./helm-chart \
  --set backend.image.tag=abc123def \
  --set frontend.image.tag=abc123def
```

This is what the CI pipeline (Chapter 8) produces — images tagged with the commit SHA. Every running pod traces back to an exact commit.

#### 6h. Resource management

Both deployments include `resources.requests` and `resources.limits`:

```yaml
resources:
  requests:
    cpu: 100m      # backend: guaranteed minimum
    memory: 128Mi
  limits:
    cpu: 500m      # backend: burst ceiling
    memory: 256Mi
```

**Why both requests and limits:**
- **Requests** tell the scheduler how much capacity a pod needs. Without requests, the scheduler packs pods without knowing their real needs — leading to node overcommit and OOM kills.
- **Limits** cap how much a pod can burst to. Without limits, one runaway pod can starve others on the same node.

**The template values are starting points.** Downstream projects should tune these based on actual profiling. The README documents how to adjust them and notes that the defaults are intentionally conservative.

#### 6i. Service Accounts

**`templates/backend-serviceaccount.yaml`** and **`templates/frontend-serviceaccount.yaml`:**

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: {{ include "fullstack-app.backend.fullname" . }}
  labels:
    {{- include "fullstack-app.backend.labels" . | nindent 4 }}
automountServiceAccountToken: false
```

Each workload gets its own ServiceAccount with `automountServiceAccountToken: false`.

**Why not use the `default` ServiceAccount:** The `default` SA in each namespace often has permissions accumulated over time (especially on older clusters or clusters with permissive RBAC). By creating a dedicated SA per workload, we start with **zero permissions**. If the application needs to call the Kubernetes API (e.g., for leader election, service discovery), RBAC roles are explicitly bound to this SA — and that decision is auditable.

**Why `automountServiceAccountToken: false`:** By default, Kubernetes mounts the SA token at `/var/run/secrets/kubernetes.io/serviceaccount/token` inside every pod. If the application doesn't call the Kubernetes API (and most web apps don't), this token is an unnecessary attack surface — a compromised container could use it to talk to the API server. Disabling automount removes that risk. Re-enable it explicitly if needed.

#### 6j. What's intentionally omitted

The following resources are **not included** in the chart. Each omission is a conscious decision, not an oversight. The chart README and the main README (Chapter 14) document each one so downstream users know what to add for their environment.

| Resource | Why it's omitted | When to add it |
| --- | --- | --- |
| **Ingress** | Depends on the cluster's ingress controller (nginx, Traefik, ALB, Istio Gateway). No sensible default exists. | When deploying to a cluster with an ingress controller. Add `templates/ingress.yaml` with annotations specific to your controller. |
| **NetworkPolicy** | Depends on the CNI plugin (Calico, Cilium, Flannel). Flannel doesn't support NetworkPolicy at all. Shipping a NetworkPolicy that the cluster silently ignores is worse than omitting it. | When the cluster CNI supports NetworkPolicy. Restrict: frontend can talk to backend; backend can talk to nothing else; deny all other ingress. |
| **HorizontalPodAutoscaler** | Requires metrics-server installed on the cluster. The template has no load profile to base scaling thresholds on. | When load patterns are understood. Scale on CPU utilization (70-80%) or custom metrics. |
| **PersistentVolumeClaim** | The template uses in-memory storage (no database). Adding a PVC implies a storage class choice that varies per cluster. | When adding a database or any stateful storage. |
| **TLS / cert-manager** | TLS termination is typically handled at the Ingress or load balancer level, not per-Service. Cert-manager configuration is cluster-specific. | When deploying with HTTPS. Usually configured alongside Ingress. |
| **PodDisruptionBudget** | Meaningful only with `replicaCount > 1`. The template defaults to 1 replica. | When running multiple replicas in production. Set `minAvailable: 1` or `maxUnavailable: 1`. |

**The rule: what's missing must be declared as missing.** A template that doesn't explain what it omits forces the user to discover gaps in production.

## Chapter 7: Pre-commit Hooks

### Type
chore

### What
Create a single `.pre-commit-config.yaml` at the repo root that orchestrates hooks for both stacks, Dockerfiles, and the Helm chart. This is the combined version of the hooks from both standalone templates, plus Helm-specific and monorepo-specific additions.

**Files:**
- `.pre-commit-config.yaml`

**Hooks to include:**

#### 1. Standard hooks (`pre-commit/pre-commit-hooks`)
- `trailing-whitespace`
- `end-of-file-fixer`
- `check-yaml` — with `args: [--allow-multiple-documents]` (Helm templates use `---` separators)
- `check-json`
- `check-added-large-files` (max 500KB)
- `check-merge-conflict`
- `debug-statements`

#### 2. Backend — Ruff (`astral-sh/ruff-pre-commit`)
- `ruff` hook — linting + import sorting
- `ruff-format` hook — code formatting
- Both scoped to backend files: `files: ^backend/`

#### 3. Backend — mypy (`pre-commit/mirrors-mypy`)
- `--ignore-missing-imports`
- `additional_dependencies`: fastapi, pydantic
- Scoped: `files: ^backend/`
- `pass_filenames: false` — mypy needs to see the full project, not individual files. The hook runs `mypy backend/src/` regardless of which backend files were staged.

#### 4. Frontend — Biome (local hook)
- Hook id: `biome-check`
- Type: `local`
- Command: `pnpm --dir frontend biome check --write --no-errors-on-unmatched --files-ignore-unknown=true`
- Stages: `pre-commit`
- File types: `ts`, `tsx`, `js`, `jsx`, `json`, `css`
- Scoped: `files: ^frontend/`
- Runs lint + format + import organization in a single pass on staged files
- `--write` auto-fixes what it can; re-stage the fixed files

**Why `local` hook type:** Biome doesn't maintain an official pre-commit mirror repo (unlike ruff which has `astral-sh/ruff-pre-commit`). The `local` hook uses the exact version pinned in `frontend/package.json` — no version drift between CI and pre-commit. The `pnpm --dir frontend` flag runs the command using the frontend directory's `node_modules`.

#### 5. Frontend — TypeScript type check (local hook)
- Hook id: `typecheck`
- Type: `local`
- Command: `pnpm --dir frontend tsc --noEmit`
- Stages: `pre-commit`
- `pass_filenames: false` — `tsc` always checks the entire project
- Scoped: `files: ^frontend/.*\.(ts|tsx)$`

#### 6. Dockerfile — Hadolint (`hadolint/hadolint` repo)
- Uses the official pre-commit repo with Docker-based execution:
  ```yaml
  - repo: https://github.com/hadolint/hadolint
    rev: v2.12.0
    hooks:
      - id: hadolint-docker
  ```
- Scoped: `files: (^backend/Dockerfile|^frontend/Dockerfile)$`
- **Why `hadolint-docker` (not `language: system`):** `language: system` requires hadolint installed on the developer's machine — fragile and hard to onboard. The `hadolint-docker` hook uses the official Docker image, which requires only Docker (already needed for this project). Portable across all developer machines.

#### 7. Helm — helm lint (local hook)
- Hook id: `helm-lint`
- Type: `local`
- Command: `helm lint helm-chart/`
- Stages: `pre-commit`
- `pass_filenames: false` — helm lint operates on the chart directory, not individual files
- Scoped: `files: ^helm-chart/`
- Requires `helm` installed locally (already needed for this project)

#### 8. Secret scanning (`gitleaks/gitleaks`)
- Catches accidentally committed secrets (.env values, API keys, tokens)
- Critical for client-facing repos: a leaked `.env` can be very costly
- No scope restriction — scans all staged files
  ```yaml
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.21.2
    hooks:
      - id: gitleaks
  ```

#### 9. Conventional Commits (`compilerla/conventional-pre-commit`)
- Stage: `commit-msg`
- Enforce `--strict` with types: `feat, fix, docs, chore, refactor, test, ci, build`
- Enforces messages like `feat: add items endpoint` and rejects `add items endpoint`
  ```yaml
  - repo: https://github.com/compilerla/conventional-pre-commit
    rev: v3.6.0
    hooks:
      - id: conventional-pre-commit
        stages: [commit-msg]
        args: [--strict, feat, fix, docs, chore, refactor, test, ci, build]
  ```

#### Monorepo scoping strategy

The key challenge of a monorepo pre-commit config is **not running Python hooks on JavaScript files and vice versa**. Each hook uses `files:` patterns to scope its execution:

| Hook | `files:` pattern | Rationale |
| --- | --- | --- |
| Standard hooks | (none — runs on all) | Trailing whitespace, YAML syntax, etc. are language-agnostic |
| Ruff, mypy | `^backend/` | Python-only tools |
| Biome, tsc | `^frontend/` | Node-only tools |
| Hadolint | `(^backend/Dockerfile\|^frontend/Dockerfile)$` | Only Dockerfiles |
| Helm lint | `^helm-chart/` | Only chart files |
| Gitleaks | (none — runs on all) | Secrets can appear anywhere |
| Conventional commits | (none — commit-msg stage) | Applies to commit messages, not files |

This ensures that a developer changing only frontend files doesn't wait for mypy to run on the backend, and vice versa.

#### Pre-commit as a system dependency

`pre-commit` is a Python tool, not a Node package. In a fullstack repo, both Python and Node developers need it. Two installation options (documented in README):

```bash
# Option A: Homebrew (macOS)
brew install pre-commit

# Option B: pip (any platform)
pip install pre-commit
```

The backend already has a Python environment via uv, but `pre-commit` is intentionally **not** added to `backend/pyproject.toml` — it's a repo-level tool, not a backend dependency. System-level install keeps it independent of both stacks.

#### Post-clone setup required
```bash
pre-commit install
pre-commit install --hook-type commit-msg
```

### Verification
```bash
# 1. Install pre-commit and git hooks
pre-commit install
pre-commit install --hook-type commit-msg

# 2. Run all hooks against existing files
pre-commit run --all-files
# All hooks should pass. On first run, pre-commit downloads and installs
# each hook's environment — this is normal and only happens once.

# 3. Verify backend hooks only run on backend files
# Touch a backend file, stage it, run:
pre-commit run ruff --files backend/src/main.py
# Should run ruff on the file

# 4. Verify frontend hooks only run on frontend files
pre-commit run biome-check --files frontend/src/App.tsx
# Should run biome on the file

# 5. Verify Hadolint runs on Dockerfiles
pre-commit run hadolint-docker --files backend/Dockerfile
pre-commit run hadolint-docker --files frontend/Dockerfile

# 6. Verify helm lint runs on chart changes
pre-commit run helm-lint --files helm-chart/values.yaml

# 7. Verify conventional commits (commit-msg stage)
git commit --allow-empty -m "bad message"
# Expected: rejected by conventional-pre-commit
git commit --allow-empty -m "chore: test conventional commits hook"
# Expected: passes
git reset HEAD~1

# 8. Verify gitleaks catches secrets
echo 'GITHUB_TOKEN=ghp_1234567890abcdef1234567890abcdef12345678' > /tmp/test-secret.txt
cp /tmp/test-secret.txt test-secret.txt
git add test-secret.txt
pre-commit run gitleaks
# Expected: blocked by gitleaks
rm test-secret.txt
```

## Chapter 8: CI Pipeline (`.github/workflows/ci.yml`)

### Type
ci

### What
The core of the template. A GitHub Actions pipeline at the root level with **7 jobs** covering both services, security scanning, Helm validation, integration testing, and registry push. This is the most complex pipeline of the three templates — a monorepo with two services and a Helm chart requires more coordination than a single-service pipeline.

**Critical:** The workflow file must live at `.github/workflows/ci.yml` in the root of the repo. GitHub Actions only discovers workflows in `.github/workflows/` at the root — files placed elsewhere are invisible to the runner.

**Files created:**
- `.github/workflows/ci.yml`
- `.trivyignore` — time-boxed CVE exception allow-list (empty by default, with format documentation in comments)

#### Workflow-level configuration

- **Triggers:** push to `main`, pull requests against `main`.
- **Environment variables** (change per project):
  - `PYTHON_VERSION` — Python version to install (e.g., `3.12`)
  - `NODE_VERSION` — Node.js version to install (e.g., `22`)
  - `APP_NAME` — Application name
  - `BACKEND_IMAGE_NAME` — Backend Docker image name
  - `FRONTEND_IMAGE_NAME` — Frontend Docker image name
- **Permissions:** least-privilege default `contents: read` at workflow level; jobs override per-need (e.g., `security-events: write` for SARIF upload, `packages: write` for registry push).

**No `PNPM_VERSION` env var.** `pnpm/action-setup@v4` reads the pnpm version directly from `frontend/package.json`'s `"packageManager"` field.

#### Pipeline structure — 7 jobs

```
backend-quality ──┐
                  ├── dockerfile-security ──┐
frontend-quality ─┘                         ├── build ── integration-test ── push
                    helm-lint ──────────────┘
```

Jobs 1 and 2 (`backend-quality`, `frontend-quality`) run **in parallel** — they have no dependency on each other. Job 3 (`dockerfile-security`) and Job 4 (`helm-lint`) also run in parallel, but both wait for the quality jobs. Job 5 (`build`) waits for security and lint jobs. Job 6 (`integration-test`) waits for build. Job 7 (`push`) waits for integration-test and only runs on main.

---

#### Job 1: `backend-quality` (Backend Code Quality & Testing)

- **Runs:** Always (push to main, PRs)
- **Working directory:** `backend/`
- **Steps:**
  1. Checkout code (`actions/checkout@v4`)
  2. Install uv (`astral-sh/setup-uv@v3`)
  3. Set up Python — `uv python install ${{ env.PYTHON_VERSION }}`
  4. Install dependencies — `uv sync --all-extras` (id: `install`)
  5. Format check — `uv run ruff format --check .` (id: `format-check`)
  6. Lint + import sorting — `uv run ruff check .` (id: `lint`)
  7. Type check — `uv run mypy src/` (id: `typecheck`)
  8. Tests with coverage — `uv run pytest --cov=src --cov-report=xml --cov-report=term --cov-fail-under=80` (id: `tests`)
  9. Upload coverage artifact — **only on GitHub** (`if: always() && !env.ACT`), `if-no-files-found: ignore`

**Fail-fast cascade:** Each step after install has `if: steps.<previous>.outcome == 'success'`. This stops the pipeline at the first failure instead of running every check and producing noisy output. Order is intentional — format is fastest, lint catches more, typecheck is slower, tests are slowest.

---

#### Job 2: `frontend-quality` (Frontend Code Quality & Testing)

- **Runs:** Always (push to main, PRs) — **in parallel with** `backend-quality`
- **Working directory:** `frontend/`
- **Steps:**
  1. Checkout code (`actions/checkout@v4`)
  2. Install pnpm (`pnpm/action-setup@v4`) — no `version` input; reads from `frontend/package.json`'s `packageManager` field. Uses `package_json_file: frontend/package.json` to point to the right file in the monorepo.
  3. Set up Node.js (`actions/setup-node@v4`) — uses `NODE_VERSION` env var, `cache: 'pnpm'`, `cache-dependency-path: frontend/pnpm-lock.yaml`
  4. Install dependencies — `pnpm install --frozen-lockfile` (id: `install`)
  5. Format check — `pnpm biome format .` (id: `format-check`)
  6. Lint + import organization — `pnpm biome lint .` (id: `lint`)
  7. Type check — `pnpm tsc --noEmit` (id: `typecheck`)
  8. Tests with coverage — `pnpm vitest run --coverage` (id: `tests`)
  9. Upload coverage artifact — **only on GitHub** (`if: always() && !env.ACT`), `if-no-files-found: ignore`

**Fail-fast cascade:** Same pattern as backend-quality. Each step gated on the previous step's success.

**Monorepo pnpm setup note:** In a monorepo, `pnpm/action-setup` needs to know where `package.json` is (it's in `frontend/`, not at root). The `package_json_file` input solves this. Similarly, `actions/setup-node`'s `cache-dependency-path` must point to `frontend/pnpm-lock.yaml` for the cache key to work correctly.

---

#### Job 3: `dockerfile-security` (Dockerfile Static Analysis)

- **Runs:** Always (including locally under `act`); **Needs:** `[backend-quality, frontend-quality]`
- **Permissions:** `contents: read`, `security-events: write`
- **Steps:**
  1. Checkout code
  2. Install Hadolint — **direct CLI download** (`curl` from GitHub releases, pinned to `v2.12.0`). Docker-based actions (`hadolint/hadolint-action`) fail under `act` due to a local image pull bug. Direct CLI invocation is consistent across GitHub and local.
  3. Hadolint — backend Dockerfile — runs the CLI twice: once with `--format sarif` (for SARIF output), once with `--failure-threshold error` (for human-readable CI logs). Only error-level findings fail the build.
  4. Hadolint — frontend Dockerfile — same approach on `frontend/Dockerfile`.
  5. Upload Hadolint SARIF to GitHub Security (`category: hadolint`) — guarded: `!cancelled() && !env.ACT && hashFiles('hadolint-results.sarif') != ''`
  6. Trivy config scan — `aquasecurity/trivy-action@master`, `scan-type: config`, `severity: CRITICAL,HIGH`, `exit-code: '1'`. Scans both Dockerfiles and YAML files for misconfigurations.
  7. Upload Trivy config SARIF to GitHub Security (`category: trivy-config`) — same guard as Hadolint upload

**Why two separate SARIF uploads:** distinct `category` values keep hadolint and trivy-config findings segregated in the Security tab for easier triage.

**Why this runs locally too:** Hadolint and Trivy config scan are static — they read files, don't touch the Docker daemon, don't need network. There is no technical reason to skip them locally.

---

#### Job 4: `helm-lint` (Helm Chart Validation)

- **Runs:** Always (including under `act`); **Needs:** `[backend-quality, frontend-quality]`
- **Steps:**
  1. Checkout code
  2. Install Helm (`azure/setup-helm@v4`)
  3. Helm lint — `helm lint helm-chart/` (id: `lint`). Validates chart syntax, catches missing values, template rendering errors.
  4. Helm template — `helm template myapp helm-chart/` (id: `template`). Renders all templates and verifies the output is valid YAML. Catches issues that `helm lint` misses (e.g., invalid resource specs that lint doesn't catch but template rendering does).
  5. Helm template with overrides — `helm template myapp helm-chart/ --set backend.image.tag=test --set frontend.image.tag=test` (id: `template-override`). Verifies that image tag overrides (used by CI/CD deployments) render correctly.

**Why a separate job (not merged into dockerfile-security):** Helm validation is conceptually different from Dockerfile scanning. Keeping it as its own job makes the Actions panel clear — a developer sees instantly whether the failure is in Docker or Helm without opening log details. It also allows independent failure: a Hadolint warning doesn't block Helm validation and vice versa.

**Why this runs locally too:** `helm lint` and `helm template` are entirely offline — they read chart files and render Go templates. No cluster, no network, no reason to skip.

---

#### Job 5: `build` (Docker Build & Image Security)

- **Runs:** Always (including under `act`); **Needs:** `[dockerfile-security, helm-lint]`
- **Permissions:** `contents: read`, `security-events: write`
- **Steps:**
  1. Checkout code
  2. Set up Docker Buildx (`docker/setup-buildx-action@v3`)

  **Backend image:**
  3. Build backend Docker image — `docker/build-push-action@v6`:
     - `context: ./backend`
     - `push: false`, `load: true`
     - `tags: ${{ env.BACKEND_IMAGE_NAME }}:${{ github.sha }}`
     - GHA cache only on GitHub, disabled under act via `${{ !env.ACT && 'type=gha' || '' }}` ternary
  4. Trivy image scan — backend — **direct CLI invocation**. Install Trivy with `aquasecurity/setup-trivy@v0.2.6` (pinned; `version: v0.69.3`). Then: `trivy image --format sarif --output trivy-backend-results.sarif --severity CRITICAL,HIGH --ignore-unfixed --ignorefile .trivyignore --exit-code 1 "$BACKEND_IMAGE_REF"`. Any CRITICAL/HIGH CVE with an upstream fix (not in `.trivyignore`) stops the pipeline.
  5. Backend container smoke test — runs the backend image, waits up to 30s for readiness, then:
     - `curl /health` and verify `"status":"healthy"`
     - POST `/items` with `{"name":"smoke-test","price":9.99}`, verify the echoed name/price
     - Extract the assigned `id` with `grep`
     - GET `/items/:id` and assert the fetched payload is **byte-identical** to the created one via `diff`
     - Dumps container logs on failure; always stops and removes the container

  **Frontend image:**
  6. Build frontend Docker image — `docker/build-push-action@v6`:
     - `context: ./frontend`
     - Same cache/tag pattern as backend, using `${{ env.FRONTEND_IMAGE_NAME }}`
  7. Trivy image scan — frontend — same CLI approach, output to `trivy-frontend-results.sarif`
  8. Frontend container smoke test — runs the frontend image, waits up to 30s, then:
     - `curl http://localhost:8080/health` and verify `"status":"healthy"`
     - `curl http://localhost:8080/` and verify the response contains `<div id="root">` (proves Vite's built `index.html` is served)
     - `curl http://localhost:8080/items` and verify the response contains `<div id="root">` (proves SPA fallback works)
     - Note: `/api/*` returns 502 because there is no backend in this isolated test — that's expected and tested in integration-test (Job 6)

  **SARIF uploads and artifacts:**
  9. Upload Trivy backend SARIF (`category: trivy-backend-image`) — guarded: `!cancelled() && !env.ACT && hashFiles('trivy-backend-results.sarif') != ''`
  10. Upload Trivy frontend SARIF (`category: trivy-frontend-image`) — same guard, `category: trivy-frontend-image`
  11. Save backend Docker image to tar — **only on main and on GitHub** (`if: github.ref == 'refs/heads/main' && !env.ACT`)
  12. Save frontend Docker image to tar — same condition
  13. Upload Docker images artifact (`actions/upload-artifact@v4`) — both tars in one artifact; `compression-level: 0`, `retention-days: 1`
  14. Cleanup built images — **act only** (`if: always() && env.ACT`). Removes tagged images to avoid local tag accumulation.

**Why Trivy CLI over `aquasecurity/trivy-action`:** Same reason as both standalone templates — when `format: sarif` is combined with `exit-code: '1'`, the action runs Trivy twice and the `trivyignores` input is only wired to the SARIF run. CLI is one invocation, deterministic, reproducible locally.

**Why separate SARIF categories per image:** Four Trivy categories total (`trivy-config`, `trivy-backend-image`, `trivy-frontend-image`) keep findings segregated in the Security tab. A developer can tell immediately whether a CVE is in the Python base image or the nginx base image.

---

#### Job 6: `integration-test` (End-to-End Stack Validation)

- **Runs:** Always (including under `act`); **Needs:** `build`
- **Steps:**
  1. Checkout code
  2. Download Docker images artifact — **only on GitHub** (`if: !env.ACT`). Under `act`, images are already in the local daemon from the `build` job running in the same Docker context.
  3. Load Docker images — `docker load` from tars (GitHub only)
  4. Tag images for docker-compose — The built images are tagged with `${{ github.sha }}`. Docker-compose expects the image names from `docker-compose.yml`. Re-tag or use a compose override to point at the SHA-tagged images.
  5. **Docker Compose up** — `docker compose up -d`. Starts both services with networking. Waits for both containers to be healthy (up to 60s).
  6. **Integration test** — the test that makes this pipeline genuinely fullstack:
     - Verify backend health: `curl http://localhost:8000/health`
     - Verify frontend health: `curl http://localhost:8080/health`
     - Verify frontend serves SPA: `curl http://localhost:8080/` contains `<div id="root">`
     - **Verify full round-trip through nginx proxy:**
       - `POST http://localhost:8080/api/items` with `{"name":"integration-test","price":42.00}` → verify 201 + returned item JSON
       - Extract the `id` from the POST response
       - `GET http://localhost:8080/api/items/:id` → verify response matches the created item
       - This proves: React build is served → nginx proxies `/api/` → backend receives the request → backend returns JSON → nginx forwards it back. The entire request path works.
     - Verify SPA fallback still works alongside API proxy: `curl http://localhost:8080/items` returns `index.html`
  7. **Dump logs on failure** — `docker compose logs` if any test step fails
  8. **Cleanup** — `docker compose down` (always)

**Why this job exists:** Jobs 1–5 validate each service in isolation. The backend smoke test proves the API works alone. The frontend smoke test proves nginx serves files and handles SPA fallback. But neither proves that **the proxy actually connects them**. The integration test is the only place where a request enters through nginx port 8080 and exits through the backend on port 8000. Without this job, a misconfigured `proxy_pass` or a wrong Service name would pass CI and break in production.

**`act` compatibility:** Under `act`, the `build` job loads images into the Docker daemon. Since `act` runs inside Docker itself, the `integration-test` job can access those same images — Docker-in-Docker shares the daemon (when using socket mount, which is `act`'s default). The only difference is that artifact download/upload is skipped; images are already present. The compose file and integration tests run identically.

**Compose file for CI:** The same `docker-compose.yml` from Chapter 5 is used. The CI overrides image names to point at the SHA-tagged builds:
```bash
BACKEND_IMAGE=${{ env.BACKEND_IMAGE_NAME }}:${{ github.sha }} \
FRONTEND_IMAGE=${{ env.FRONTEND_IMAGE_NAME }}:${{ github.sha }} \
docker compose up -d
```
This requires the `docker-compose.yml` to use `image:` fields with variable interpolation (e.g., `image: ${BACKEND_IMAGE:-template-fullstack-backend}`), falling back to the default name for local development. Alternatively, a `docker-compose.ci.yml` override file can be used.

#### Job 7: `push` (Registry Push)

- **Runs:** Only on push to main (`if: github.ref == 'refs/heads/main'` at job level). Under `act`, the job starts but every step is skipped via per-step `if: ${{ !env.ACT }}` guards. `env.ACT` is **not** valid in a job-level `if` — act's schema rejects it, and GitHub's docs restrict job-level `if` to the `github`, `needs`, `vars`, and `inputs` contexts — so the local-skip guard must live at step level.
- **Needs:** `integration-test`
- **Permissions:** `contents: read`, `packages: write` (needed when pushing to ghcr.io)
- **Steps:** (every step carries `if: ${{ !env.ACT }}`)
  1. Verify registry credentials — prints whether `REGISTRY_USERNAME`, `REGISTRY_TOKEN`, and `REGISTRY_URL` are configured, and **hard-fails with `exit 1` and an `::error::` annotation** if either secret is missing. Without this, a missing secret fails later with a confusing "Cannot perform an interactive login from a non TTY device" error.
  2. Download Docker images artifact (`actions/download-artifact@v4`)
  3. Load Docker images — `docker load` from both tars
  4. Log in to registry — `docker/login-action@v3`. Empty `registry` input → defaults to Docker Hub.
  5. Tag and push — backend: `docker tag` + `docker push`. Frontend: same. Target ref computed at step level: `<REGISTRY_URL>/<image>:<sha>` when `REGISTRY_URL` is set, otherwise `<image>:<sha>` (Docker Hub). Exposes `backend_ref` and `frontend_ref` as step outputs.
  6. Registry push summary — on success, writes a markdown table (Images / Commit / Registry) to `GITHUB_STEP_SUMMARY`. Lists both backend and frontend image refs. Guarded by `success() && !env.ACT`.

**Push exactly what was scanned:** The `build` job saves both images as tar artifacts after Trivy scanning and smoke testing. The `push` job loads and re-tags those exact bytes. No rebuild. The images going to the registry are guaranteed identical to what was scanned and integration-tested.

**Registry-agnostic design:** Same secret names as both standalone templates: `REGISTRY_USERNAME`, `REGISTRY_TOKEN`, `REGISTRY_URL`. The README (Chapter 14) documents configuration for Docker Hub, ghcr.io, AWS ECR (OIDC), ACR, and Harbor/Nexus/Artifactory.

## Chapter 10: Branch Protection on `main`

### Type
chore

### What
Configure GitHub branch protection rules on `main` so direct pushes are blocked and every change goes through a PR with green CI.

**When to apply:** After Chapter 9's first GitHub run has **completed green**. GitHub's "Require status checks to pass before merging" dropdown is populated from historical workflow runs — a check that has never executed is not selectable by name. Configuring protection before the first run means either the dropdown is empty or you save the rule with no checks selected and protection is a no-op. The correct order is always: push → first run completes green → configure protection and select the now-visible check names.

**Rules to configure** (Settings → Branches → Add branch protection rule → Branch name pattern: `main`):

| Rule | Why |
| --- | --- |
| **Require a pull request before merging** | No direct pushes to `main`. |
| **Require approvals** (minimum 1) | At least one reviewer signs off. Small teams: team lead. Larger teams: rotation. |
| **Require status checks to pass before merging** | Select every PR-triggered job: `backend-quality`, `frontend-quality`, `dockerfile-security`, `helm-lint`, `build`, `integration-test`. The `push` job runs only on main, so it will not appear in the dropdown — this is expected. |
| **Require branches to be up to date before merging** | PR must be synced with latest `main` before merge. Prevents two independently-green PRs from breaking `main` when combined. Trade-off: on high-traffic repos this causes a rebase treadmill. For a fullstack template used by small teams this is acceptable. |
| **Do not allow force pushes** | Never on `main`. Protects commit history. |
| **Do not allow deletions** | Never on `main`. Prevents accidental branch deletion. |
| **Require linear history** (recommended) | Forces squash or rebase merges; no merge commits. History stays readable with `git log --oneline`. Pairs naturally with up-to-date requirement. |

**Applies to two places:**
- This template repo itself — configure once as part of shipping the template.
- Every new repo created from this template — GitHub does **not** carry branch protection rules across template copies. The README (Chapter 14) must document this as a mandatory post-bootstrap step for downstream users.

**Optional automation (out of scope for v1):** These rules can be codified via `gh api repos/:owner/:repo/branches/main/protection` calls or the Terraform `integrations/github` provider so they're reproducible across repos created from this template.

### Verification
```bash
# 1. Direct push to main must be rejected
git checkout main
git commit --allow-empty -m "test: direct push"
git push origin main
# Expected: remote: error: GH006: Protected branch update failed for refs/heads/main

# 2. Force-push to main must be rejected
git push --force origin main
# Expected: remote: error: GH003: Sorry, force-pushing to main is not allowed

# 3. Open a PR with a deliberately failing check — the "Merge" button must be disabled
#    (manual verification in the GitHub UI)

# 4. Open a PR that is behind main — the UI must show "This branch is out-of-date
#    with the base branch" and block merge until updated (manual verification)
```

## Chapter 12: Dependabot Configuration

### Type
ci

### What
Create `.github/dependabot.yml` for automated dependency updates across all four dependency ecosystems in the monorepo.

**File:** `.github/dependabot.yml`

**Ecosystems to track (all weekly):**

1. **docker** (`backend/`) — Bumps base image digests in `backend/Dockerfile` (Python, uv). Catches new patch versions and security fixes.

2. **docker** (`frontend/`) — Bumps base image digests in `frontend/Dockerfile` (Node, nginx). Separate entry because Dependabot needs one `directory` per ecosystem entry.

3. **pip** (`backend/`) — Bumps Python dependencies in `backend/pyproject.toml`. Dependabot understands uv lockfiles (`uv.lock`).

4. **npm** (`frontend/`) — Bumps Node dependencies in `frontend/package.json`. Dependabot understands pnpm lockfiles (`pnpm-lock.yaml`) natively since 2023.

5. **github-actions** (root `/`) — Bumps action versions in `.github/workflows/ci.yml`. Catches security patches in third-party actions.

```yaml
version: 2
updates:
  # Backend Docker base images (Python, uv)
  - package-ecosystem: "docker"
    directory: "/backend"
    schedule:
      interval: "weekly"

  # Frontend Docker base images (Node, nginx)
  - package-ecosystem: "docker"
    directory: "/frontend"
    schedule:
      interval: "weekly"

  # Backend Python dependencies
  - package-ecosystem: "pip"
    directory: "/backend"
    schedule:
      interval: "weekly"

  # Frontend Node dependencies
  - package-ecosystem: "npm"
    directory: "/frontend"
    schedule:
      interval: "weekly"

  # GitHub Actions versions
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
```

**Why five entries instead of one:** Dependabot requires one entry per `(ecosystem, directory)` pair. The monorepo has Dockerfiles in two directories and package manifests in two directories. The GitHub Actions entry always targets `/` because workflows live at `.github/workflows/` relative to the root.

**Why weekly:** Daily is too noisy for a template repo (multiple PRs per week for minor bumps). Monthly is too slow for security patches. Weekly is the standard balance — a digest bump PR that fixes a CVE lands within a week.

**Why this matters:** All external dependencies in this template are pinned (Docker image digests, Python/Node package versions, GitHub Action versions). Pinning prevents silent changes but creates a maintenance burden — without automation, pinned versions rot. Dependabot closes the loop: it watches upstream for new versions and opens PRs with the bump. The CI pipeline validates the bump. A human reviews and merges. Dependencies stay current without manual version-chasing.

### Verification
```bash
# Verify file exists and is valid YAML
test -f .github/dependabot.yml && echo "OK: dependabot.yml exists"
python3 -c "import yaml; yaml.safe_load(open('.github/dependabot.yml'))" && echo "OK: valid YAML"
```

## Chapter 14: README.md

### Type
docs

### What
Comprehensive template documentation. The README is the entry point for anyone using this template — a colleague, a client, or yourself in six months. #### 13. Design decisions

| Decision | Why |
| --- | --- |
| **Monorepo** over separate repos | Frontend and backend are deployed together, versioned together, and tested together. A monorepo makes the integration test natural and avoids cross-repo coordination. |
| **Helm** over raw K8s manifests | Templating, versioning, rollback with `helm rollback`, configurable values per environment. Raw manifests require `envsubst` or `kustomize` for the same flexibility. |
| **Docker Compose AND Helm** | Different tools for different contexts. Compose for local dev (no cluster needed). Helm for K8s deployment. Same Dockerfiles and nginx.conf power both. |
| **nginx as API reverse proxy** | The frontend container handles both static serving and API proxying. One entry point, one port, one Ingress rule. No need for the client to know the backend exists. |
| **Vite dev proxy mirroring nginx** | Frontend code always calls `/api/*`. In dev, Vite proxies to `localhost:8000`. In prod, nginx proxies to the backend Service. Same code, different proxy layer. |
| **`envsubst` for dynamic nginx config** | Simpler than ConfigMap-mounted snippets. The backend URL is injected at container startup, not baked into the image. |
| **FastAPI + uv + ruff** (backend) | Same reasoning as template-backend-python. Fast, modern, single-tool-per-concern. |
| **React + Vite + pnpm + Biome** (frontend) | Same reasoning as template-frontend-react. Fast, strict, single-tool-per-concern. |
| **Separate quality jobs** (parallel) | Backend and frontend quality checks run in parallel. A backend lint failure doesn't block frontend tests. Faster feedback, clearer attribution. |
| **Integration test job** | The only place that proves the services work together. Smoke tests validate each service in isolation; integration-test validates the proxy chain. |
| **7 jobs, not 4** | More jobs = clearer Actions panel. A developer sees instantly which concern failed (backend code? frontend code? Dockerfile? Helm? build? integration? push?) without opening logs. |
| **Digest-pinned Docker images** | Immutable builds. Dependabot manages the update cadence. |
| **Non-root containers** | Principle of Least Privilege. PSA `restricted` compliance. |
| **No `tag: latest`** | Traceability. Every running pod maps to an exact commit SHA. |
| **Surgical `act` skip logic** | Step-level, not job-level. Everything that validates the artifact runs locally. Only external side effects are skipped. |
